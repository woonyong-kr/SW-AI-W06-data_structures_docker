# AI 활용 원칙 (Week 6)

기간: 2026-04-03 ~ 2026-04-09

이번 주는 C, 포인터, 동적 할당, 기본 자료구조 구현이 핵심이라서 AI도 그 기준에 맞춰 사용한다.
목표는 빠르게 푸는 것보다 팀원 전체가 설명할 수 있게 이해하는 것이다.

## 기본 원칙

- AI는 바로 답을 받아 적는 용도보다 질문하고 정리하는 용도로 먼저 쓴다.
- 1차적으로는 질문 위주로 쓰고, 그래도 막히면 답이나 코드 예시를 요청해도 된다.
- AI가 준 답이나 코드는 왜 그렇게 짰는지 반드시 설명할 수 있어야 한다.
- 설명이 안 되면 아직 내 코드가 아니라는 기준으로 다시 본다.
- 코어타임에는 AI 도움을 받은 내용도 직접 설명하거나 발표한다.
- C 프로젝트이므로 C++ 문법이나 STL 같은 건 쓰지 않는다.
- AI가 준 코드는 그대로 쓰지 않고 직접 실행하고 확인한 뒤 반영한다.

## 과제별 기준

| 과제 | 사용 기준 |
|------|-----------|
| C언어 문법 학습 | 익숙하지 않은 사람만 진행. 총 8시간 넘기지 않음 |
| Data Structures | 질문 위주 사용. 포인터, `malloc/free`, 디버깅 포인트 위주로 물어봄 |
| 수요 코딩회 SQL 처리기 | 구현 속도를 위해 적극 활용 가능. 대신 핵심 로직은 설명 가능해야 함 |

## C언어 학습 쪽

- 문법 전부 끝내는 게 목표는 아님.
- 과제 진행에 필요한 만큼만 익히고, 모르는 건 하면서 계속 채운다.
- 우선순위는 `문법 -> gcc 사용 -> 포인터 -> malloc/free -> 과제 구현` 순서로 잡는다.
- 구조체, 포인터, 문자열, 파일 입출력처럼 바로 쓰는 내용부터 본다.

## Data Structures 쪽

- 막히면 혼자 오래 끌지 말고 팀원이랑 먼저 본다.
- 그 다음 AI에는 이유를 묻는 식으로 질문한다.
- 예시:
  `이 포인터가 왜 필요한지`
  `여기서 free를 누가 해야 하는지`
  `세그폴트가 날 수 있는 지점이 어디인지`
  `기존 코드를 어디까지 수정해야 하는지`
- 정답만 받기보다 반례, 디버깅 포인트, 메모리 해제 순서를 같이 확인한다.

## 수요 코딩회 쪽

- 이번 과제는 학습보다 구현 우선이라 AI를 적극적으로 써도 된다.
- 다만 아래는 팀원이 직접 설명 가능해야 한다.

| 항목 | 설명 가능 기준 |
|------|----------------|
| SQL 파싱 | `INSERT`, `SELECT`를 어떻게 구분하는지 설명 |
| 실행 흐름 | 파싱 결과가 파일 읽기/쓰기 로직으로 어떻게 이어지는지 설명 |
| 저장 포맷 | 왜 그 포맷을 골랐는지 설명 |
| 테스트 | 어떤 입력으로 검증했는지 설명 |

## 하지 말 것

- 설명 못 하는 코드를 그대로 제출하지 않기
- 동작만 하면 됐다고 넘기지 않기
- 포인터, 동적 할당, 파일 I/O는 AI 답을 그대로 믿지 않기
- 메모리 누수나 이중 해제는 `gdb`, `valgrind`, 직접 테스트로 다시 보기

## 코어타임 체크

- 오늘 AI 도움받은 내용 중 하나는 직접 설명하기
- 포인터나 동적 할당 관련 개념 하나는 팀에 공유하기
- SQL 처리기 쪽은 핵심 로직이나 테스트 결과 하나는 발표하기
