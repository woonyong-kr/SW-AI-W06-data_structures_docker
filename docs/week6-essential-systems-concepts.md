# Week 6 문제 해결에 필요한 핵심 개념 정리

이 문서는 이번 주 학습 주제, 현재 저장소의 C 소스코드, 그리고 `컴퓨터 시스템` 1장, 3장, 9장에서 실제 문제 풀이에 바로 필요한 개념만 추려서 정리한 문서다.

목표는 세 가지다.

- Linked List, Stack, Queue, Tree, BST 문제를 풀 때 왜 이런 코드가 필요한지 이해하기
- 포인터와 동적 할당 때문에 생기는 실수를 줄이기
- 디버깅할 때 어디를 봐야 하는지 빠르게 판단하기

## 1. 이 저장소에서 반복되는 코드 패턴

이 저장소 문제들은 겉보기엔 주제가 달라도 실제로는 비슷한 패턴을 계속 쓴다.

### 1-1. Linked List 기본 뼈대

`Data-Structures/Linked_List/Q1_A_LL.c`를 보면 기본 구조가 거의 다 들어 있다.

- `ListNode`는 `item`과 `next`를 가진다.
- `LinkedList`는 `head`와 `size`를 가진다.
- 실제 삽입/삭제는 보조 함수 `insertNode`, `removeNode`, `findNode`를 재사용한다.

즉, 많은 문제는 "새로운 자료구조"를 만드는 게 아니라 아래 중 하나다.

- 노드를 어디에 끼워 넣을지 정하기
- 노드를 직접 움직일지, 새로 만들어 복사할지 정하기
- `head`가 바뀌는 경우를 따로 처리하기
- `size`를 언제 바꿔야 하는지 맞추기

### 1-2. Stack, Queue는 결국 Linked List 위의 규칙이다

`Data-Structures/Stack_and_Queue/Q1_C_SQ.c`, `Q2_C_SQ.c`를 보면:

- Queue는 `insertNode(..., size, value)`와 `removeNode(..., 0)`를 이용한다.
- Stack은 `insertNode(..., 0, value)`와 `removeNode(..., 0)`를 이용한다.

즉 핵심은 자료구조 이름보다 "어느 쪽에서 넣고 어느 쪽에서 빼는가"다.

- Queue: FIFO
- Stack: LIFO

문제를 풀 때는 "이 값을 보존해야 하나", "원래 순서를 유지해야 하나", "임시 자료구조가 더 필요한가"를 먼저 본다.

### 1-3. Tree, BST는 포인터가 2개 이상인 노드 문제다

`Data-Structures/Binary_Tree/Q1_E_BT.c`, `Data-Structures/Binary_Search_Tree/Q1_F_BST.c`를 보면:

- Tree 노드는 `left`, `right`를 가진다.
- BST는 Tree이면서 추가로 `left < root < right` 규칙이 있다.
- 재귀로 풀 수 있는 문제와, stack/queue를 써서 반복문으로 풀 수 있는 문제가 섞여 있다.

그래서 트리 문제는 보통 두 갈래다.

- 현재 노드 기준으로 값을 계산하는 문제
- 순회 순서 자체가 중요한 문제

## 2. 이번 주 필수 C 개념

이번 주 과제 기준으로 C 문법 전체가 필요한 건 아니다. 아래가 먼저다.

### 2-1. 구조체

이 저장소 거의 모든 문제는 구조체로 자료구조를 표현한다.

```c
typedef struct _listnode{
    int item;
    struct _listnode *next;
} ListNode;
```

중요한 점:

- 구조체 안에서 자기 자신 타입을 바로 값으로 둘 수는 없고 포인터로 둔다.
- `ListNode *next`는 "다음 노드의 주소"다.
- 노드 연결을 바꾼다는 건 결국 주소를 다시 대입하는 일이다.

### 2-2. 포인터

이번 주 핵심이다.

- `p`는 주소를 저장하는 변수
- `*p`는 그 주소가 가리키는 실제 값
- `&x`는 `x`의 주소
- `node->next`는 `(*node).next`와 같다

꼭 구분해야 하는 것:

- 포인터 변수 자체를 바꾸는 것
- 포인터가 가리키는 대상의 내용을 바꾸는 것

예를 들어:

- `cur = cur->next;` 는 "현재 포인터를 다음으로 이동"
- `cur->next = newNode;` 는 "노드 연결 자체를 변경"

### 2-3. 포인터를 함수 인자로 넘길 때

이 부분을 틀리면 함수가 동작하지 않는 경우가 많다.

- `LinkedList *ll`: 리스트 내부 필드 수정 가능
- `ListNode *head`: head가 가리키는 노드 내용은 볼 수 있지만 head 자체를 바꿔도 호출자에는 반영 안 될 수 있음
- `ListNode **ptrHead`: head 자체를 바꿔야 할 때 사용

예를 들어 아래 같은 함수는 `head` 변경이 필요해서 이중 포인터가 자주 나온다.

- 리스트 앞 노드 삽입
- 최대값을 맨 앞으로 이동
- 재귀 뒤집기

그래서 `Q6_A_LL.c`, `Q7_A_LL.c` 류 문제는 `ListNode **`를 이해해야 편하다.

### 2-4. 동적 할당

노드는 런타임에 개수가 정해지므로 보통 `malloc`으로 만든다.

```c
ListNode *newNode = malloc(sizeof(ListNode));
```

중요한 점:

- `sizeof(ListNode)` 만큼 정확히 할당해야 한다.
- `malloc`이 성공하면 사용 전에 필드를 초기화해야 한다.
- 더 이상 안 쓰는 노드는 `free`해야 한다.
- `free`한 포인터를 다시 쓰면 안 된다.

이번 저장소에서 흔한 실수:

- 노드를 빼고 연결은 바꿨는데 `free`를 안 함
- `free` 먼저 해버리고 `next`를 읽으려고 함
- head가 바뀌는 상황인데 지역 포인터만 바꾸고 끝냄
- `size`는 줄였는데 연결은 안 바꿈

## 3. 문제 풀이에 필요한 시스템 관점

책 1장, 3장, 9장은 이번 주 과제에 바로 연결된다.

## 3-1. 1장 관점: "프로그램은 시스템 전체 안에서 돈다"

1장은 시스템 전체를 크게 보는 장이다. 이번 주에 필요한 핵심만 가져오면 아래다.

### 왜 필요한가

- 내가 쓴 `.c` 파일이 어떻게 실행 파일이 되는지 알아야 컴파일 에러를 덜 무서워한다.
- 메모리, OS, CPU, 컴파일러가 다 연결돼 있다는 감각이 있어야 디버깅 방향을 잡을 수 있다.

### 이번 주에 바로 연결되는 포인트

- C 소스코드는 컴파일되어 실행 파일이 된다.
- 프로그램은 메모리 위에서 실행된다.
- 시스템은 추상화 계층으로 되어 있어서, 우리가 보는 포인터/함수 호출도 결국 하드웨어/OS 위에서 돌아간다.
- 좋은 프로그래머는 문법만 아는 사람이 아니라 메모리와 실행 흐름까지 보는 사람이다.

### 실전 연결

- `gcc` 경고를 무시하지 않기
- 실행은 되는데 이상하면 입력, 메모리, 포인터, 함수 호출 순서까지 같이 보기
- 자료구조 문제도 결국 메모리 위에 구조를 만드는 일이라고 생각하기

## 3-2. 3장 관점: "함수 호출, 포인터, 배열, 구조체는 메모리 배치와 연결된다"

이번 주에 가장 직접적으로 중요한 장이다.

### 왜 필요한가

- 포인터가 왜 위험한지
- 재귀 호출이 왜 동작하는지
- 구조체와 배열 접근이 왜 그런 문법인지
- 함수 인자 전달이 왜 값 복사처럼 보이는지

이걸 이해하는 데 도움을 준다.

### 이번 주에 바로 필요한 핵심

#### 함수 호출과 스택 프레임

함수를 호출하면 스택에 호출 정보와 지역변수 공간이 쌓인다.

그래서:

- 지역변수 주소를 잘못 오래 들고 있으면 위험하다
- 재귀는 호출할 때마다 새 프레임이 생긴다
- 재귀 깊이가 깊어지면 스택 사용량이 늘어난다

Linked List 재귀 뒤집기나 Tree 재귀 순회가 왜 동작하는지 이해하려면 이 관점이 필요하다.

#### 구조체/배열/포인터 접근

기계 수준에서는 결국 메모리 주소 계산이다.

- `a[i]`는 주소 계산 후 접근
- `node->left`도 구조체 기준 오프셋 접근
- `*(p + i)`와 배열 접근은 같은 계열이다

그래서 포인터를 잘못 한 칸만 밀어도 엉뚱한 메모리를 볼 수 있다.

#### 값 전달과 포인터 전달

C는 기본적으로 값 전달이다.

즉:

- `foo(x)` 에서 `x` 자체는 복사된다
- 하지만 `x`가 포인터면 "주소값"이 복사된다
- 그래서 포인터가 가리키는 대상은 수정될 수 있다

이게 `LinkedList *ll`가 함수 밖 리스트를 바꿀 수 있는 이유다.

#### 재귀와 반복

Tree 문제는 재귀가 자연스럽고, BST 순회는 stack/queue를 써서 반복으로도 푼다.

필요한 판단:

- 하위 문제 구조가 그대로 반복되면 재귀
- 순회 순서 제어가 중요하거나 명시적 자료구조가 필요하면 반복 + stack/queue

## 3-3. 9장 관점: "가상 메모리와 동적 메모리 할당"

이번 주 과제에서 직접 체감되는 장이다.

### 왜 필요한가

- `malloc/free`가 왜 필요한지
- stack 변수와 heap 변수가 어떻게 다른지
- 세그폴트가 왜 나는지
- 메모리 누수가 뭔지

를 이해하게 해준다.

### 꼭 알아야 하는 메모리 구역

- text: 코드 영역
- data/bss: 전역/정적 데이터
- stack: 함수 호출, 지역변수
- heap: `malloc`으로 직접 받은 공간

이번 주 문제는 노드를 대부분 heap에 만든다.

즉:

- 함수 끝나도 노드가 살아 있어야 하므로 heap이 필요하다
- 지역변수로 만든 노드를 연결해두면 함수가 끝난 뒤 위험해진다

### 동적 할당에서 꼭 챙길 것

- 누가 할당하는지
- 누가 해제하는지
- 한 번 해제한 노드를 다시 읽지 않는지
- 리스트/트리 전체 종료 시 정리 함수가 있는지

이 저장소에서도 `removeAllItems`, `removeAll`, `removeAllItemsFromQueue`, `removeAllItemsFromStack` 같은 정리 함수가 계속 나온다.

이건 단순 편의가 아니라 메모리 정리 책임을 분리해 둔 것이다.

### 세그멘테이션 폴트가 자주 나는 상황

- `NULL`인데 `node->next` 접근
- 이미 `free`한 노드 접근
- head가 `NULL`인데 첫 노드가 있다고 가정
- 삭제 후 연결을 잘못 이어서 끊어진 노드를 다시 접근

## 4. 폴더별로 어떤 개념이 특히 중요한가

## 4-1. Linked List

핵심 개념:

- 단일 연결 리스트에서 이전 노드를 같이 추적하는 법
- head 변경 처리
- 노드 삽입/삭제 시 연결 순서
- size 유지
- 이중 포인터가 필요한 경우

자주 나오는 문제 유형:

- 정렬 삽입
- 번갈아 병합
- 조건에 맞는 노드를 뒤로 보내기
- 리스트 분할
- 최대값 앞으로 이동
- 재귀 뒤집기

문제 풀 때 체크:

1. 빈 리스트인가
2. 노드 1개인가
3. 첫 노드가 바뀌는가
4. 기존 노드를 재배치하는가, 새 노드를 생성하는가
5. 마지막 노드 처리에서 `next = NULL`이 필요한가

## 4-2. Stack / Queue

핵심 개념:

- Stack: 한쪽에서 push/pop
- Queue: 뒤로 넣고 앞으로 뺌
- 기존 리스트를 복사해 만들지, 노드를 옮길지 결정
- 임시 stack/queue 사용으로 순서를 보존할지 뒤집을지 판단

문제 풀 때 체크:

1. 원래 순서를 유지해야 하는가
2. 값을 제거하면서도 자료구조를 원상복구해야 하는가
3. 검사만 할지 실제로 구조를 바꿀지

예:

- `removeOddValues`, `removeEvenValues`: 순회하면서 조건 필터링
- `reverse`, `recursiveReverse`: 순서 변환
- `balanced`: 짝 맞추기 상태 추적

## 4-3. Binary Tree

핵심 개념:

- 재귀 순회
- base case 설정
- 현재 노드 처리와 하위 트리 처리 순서
- NULL 자식 처리

자주 묻는 패턴:

- 높이 계산
- 조건 개수 세기
- 합 구하기
- mirror
- 최소값/특정 조건 확인
- 구조 동일성 비교

트리 문제 기본 틀:

```c
if (node == NULL) {
    return ...;
}

left_result = solve(node->left);
right_result = solve(node->right);

return combine(node, left_result, right_result);
```

## 4-4. Binary Search Tree

핵심 개념:

- BST 성질: 왼쪽 < 루트 < 오른쪽
- 순회와 BST 성질을 연결해서 생각하기
- iterative traversal에서 stack/queue 사용

이번 폴더에서 중요한 건:

- level-order traversal은 queue
- inorder / preorder / postorder iterative는 stack

즉, "트리 문제"와 "스택/큐 문제"가 합쳐진 형태다.

## 6. 이번 주 문제 해결용 최소 체크리스트

- 구조체 정의를 정확히 읽었는가
- head, size, next/left/right가 언제 바뀌는지 적어봤는가
- 빈 구조와 원소 1개 케이스를 따로 처리했는가
- 기존 노드를 움직이는지 새 노드를 만드는지 정했는가
- `malloc` 뒤 초기화했는가
- 삭제 시 `free`와 연결 변경 순서를 점검했는가
- 함수가 끝난 뒤에도 살아야 하는 데이터는 heap에 있는가
- 재귀면 base case가 명확한가
- iterative traversal이면 stack/queue 동작을 손으로 따라가 봤는가
- 테스트 입력을 직접 몇 개 넣어봤는가
