# AI 활용 원칙 (Week 6)

> 작성 시점: 주차 시작 시
>
> 적용 기간: 2026-04-03 ~ 2026-04-09

이번 주는 학습 과제(금~화)와 수요 코딩회(수)의 성격이 다르므로, 과제 목적에 맞게 AI 사용 범위를 구분합니다.

## 과제별 AI 사용 범위

| 과제 | AI 사용 | 필수 조건 |
|------|---------|-----------|
| C언어 문법 학습 | 조건부 허용 | C 문법에 익숙하지 않은 사람만 진행하고, 총 학습 시간은 8시간 이내로 제한 |
| Data Structures 구현 | 학습 보조 위주 | 1차적으로는 질문 중심으로 사용하고, 답 전체 요청은 마지막 수단으로 사용 |
| 수요 코딩회 SQL 처리기 구현 | 적극 활용 | 구현 속도를 높이기 위해 AI 활용 가능. 단, 핵심 로직은 팀원이 직접 설명하고 테스트로 검증해야 함 |

## 우리 팀 기본 원칙

- AI는 정답 생성기가 아니라 학습과 구현을 돕는 도구로 사용합니다.
- 1차적으로는 질문 위주로 사용하고, 바로 정답이나 완성 코드를 요구하지 않습니다.
- 이해가 막히는 경우에는 AI에게 답을 요청할 수 있습니다.
- AI가 제안한 답이나 코드가 있다면, 왜 그렇게 작성했는지와 왜 그 방식이 필요한지를 반드시 설명할 수 있어야 합니다.
- 코어타임에는 AI에게 받은 설명이나 코드 중 핵심 내용 하나 이상을 직접 발표하고 공유합니다.
- AI가 작성한 코드는 그대로 제출하거나 머지하지 않고, 직접 실행·수정·검증한 뒤 사용합니다.
- C++이 아니라 C로 구현해야 하므로, AI가 C++ 문법이나 STL을 제안하면 사용하지 않습니다.

## 이번 주 권장 사용 방식

### 1. C언어 학습

- 문법 전체를 완주하는 것이 아니라 과제를 진행할 만큼 익히는 것을 목표로 합니다.
- 모르는 문법이 나올 때 `개념 설명`, `짧은 예제`, `주의할 점` 정도를 묻는 식으로 사용합니다.
- 포인터, 구조체, 함수 선언, 배열, 문자열 처리처럼 과제에 바로 연결되는 내용 위주로 질문합니다.

### 2. Data Structures 구현

- 막히는 부분은 먼저 팀원과 논의하고, 그 다음 AI에게 질문합니다.
- 질문은 `이 포인터가 왜 필요한지`, `malloc/free를 어디서 해야 하는지`, `이전 코드에서 무엇을 수정해야 하는지`처럼 이유 중심으로 합니다.
- 단순 정답 요청보다 `반례`, `디버깅 포인트`, `메모리 해제 순서`, `세그멘테이션 폴트 원인`을 묻는 방식으로 활용합니다.

### 3. 수요 코딩회 SQL 처리기 구현

- 이번 과제는 "학습"보다 "구현"이 우선이므로 AI를 적극 활용합니다.
- 다만 아래 항목은 팀원이 직접 설명할 수 있어야 합니다.

| 설명 가능해야 하는 항목 | 최소 설명 기준 |
|------------------------|----------------|
| SQL 파싱 구조 | 입력을 어떤 단위로 나누고, 어떤 조건으로 `INSERT`/`SELECT`를 구분하는지 설명 |
| 실행 흐름 | 파싱 결과가 어떻게 파일 읽기/쓰기 로직으로 연결되는지 설명 |
| 데이터 저장 포맷 | CSV/binary/custom 중 무엇을 선택했고 왜 그렇게 정했는지 설명 |
| 테스트 전략 | 어떤 입력/엣지 케이스를 검증했고 결과가 무엇이었는지 설명 |

## 추천 질문 예시

- `C에서 포인터 변수와 포인터가 가리키는 값의 차이를 예제로 설명해줘`
- `Linked List에서 노드를 삭제할 때 free 순서를 왜 이렇게 잡아야 해?`
- `malloc으로 할당한 메모리를 누가 소유하는지 함수 단위로 어떻게 정리하면 좋을까?`
- `SQL 문장을 tokenizing -> parsing -> execution으로 나누면 어떤 장점이 있어?`
- `INSERT/SELECT를 파일 기반으로 구현할 때 테스트 케이스를 어떻게 구성하면 좋을까?`

## 금지 및 주의 사항

- 설명할 수 없는 AI 생성 코드를 그대로 사용하지 않습니다.
- 코어타임 발표 전에 이해되지 않은 코드를 "일단 동작하니까"라는 이유로 넘기지 않습니다.
- 포인터, 동적 할당, 파일 I/O처럼 위험한 영역은 AI 답변을 그대로 믿지 않고 직접 실행 결과로 검증합니다.
- 메모리 누수, 이중 해제, 잘못된 포인터 참조 가능성은 `gdb`, `valgrind`, 자체 테스트로 다시 확인합니다.

## 코어타임 확인 항목

- 오늘 AI에게서 도움받은 내용 중 직접 설명 가능한 개념 1개 이상 공유
- 포인터 또는 동적 할당 관련 핵심 개념 1개 이상 설명
- SQL 처리기 핵심 로직 또는 테스트 결과 1개 이상 발표
